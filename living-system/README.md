# Living System

This folder defines how the repository should evolve over time.

The goal is simple:

Do not let the skill stay frozen at "version 1 understanding."
Every time strong real copy is produced, the repository should learn from it.

## What Gets Fed Back

Future real outputs should be fed back when they contain any of the following:

- a particularly strong opening
- a cleaner version of an old argument
- a new title pattern
- a better way to explain a familiar topic
- a memorable closing line
- a new miniature story that proves a structure
- a new high-quality platform-specific format

Do not feed back weak or generic copy just because it is recent.

## The Feedback Loop

### Step 1: Save the piece

If a piece feels strong and recognizably Xixi, store it in `corpus/sample-index.md`.

Log:

- date
- topic
- format
- why it was strong
- what it added to the voice system

### Step 2: Extract patterns

After storing a strong piece, extract:

- new sentence patterns
- new framing moves
- new openings
- new endings
- new structural transitions

Store these in `corpus/extracted-patterns.md`.

### Step 3: Update the right reference file

Depending on what the new piece taught, update one or more of:

- `references/sentence-bank.md`
- `references/voice-corpus.md`
- `references/before-after-pack.md`
- `references/case-library.md`
- `references/platform-playbooks.md`

### Step 4: Log the change

Record the update in `corpus/update-log.md` so the repo evolves intentionally.

## What Not To Do

- do not dump raw drafts without explanation
- do not add low-quality pieces
- do not overfit the whole system to one lucky paragraph
- do not confuse novelty with voice quality

## Principle

The repository should grow by compression, not by clutter.

Each new piece should help answer:

- what does Xixi do repeatedly
- what does Xixi do better now than before
- what new pattern is now stable enough to preserve
